LLM 기반 **DBA Agent**를 만들 때 핵심은 **LLM 자체보다 “Agent Harness Engineering”** 입니다.
즉, LLM이 **DB를 직접 건드리지 않고**, **안전한 실행 레이어(Harness)** 를 통해 DBA 작업을 수행하도록 설계해야 합니다.

특히 PostgreSQL 기반이라면 아래 **7개 영역을 사전에 설계**해야 합니다.

---

# 1. Agent Harness Architecture 설계

LLM이 직접 SQL을 실행하면 **치명적인 사고**가 발생합니다.
따라서 **Execution Harness Layer**가 필요합니다.

### 기본 구조

```
User
 ↓
Chat Interface
 ↓
LLM Reasoning Layer
 ↓
Agent Harness Layer
 ├─ Tool Router
 ├─ Safety Guard
 ├─ Query Planner
 ├─ Execution Engine
 ↓
PostgreSQL
```

Harness의 핵심 역할

| 역할                 | 설명                    |
| ------------------ | --------------------- |
| Tool Routing       | LLM이 어떤 Tool을 호출할지 결정 |
| SQL Validation     | 위험한 쿼리 차단             |
| Permission Control | DBA 권한 제한             |
| Plan Simulation    | 실행 전에 영향 분석           |
| Rollback Control   | 실패 시 복구               |

---

# 2. DBA Capability Scope 정의 (Agent Skill Set)

Agent가 수행할 **DBA 업무 범위**를 먼저 정의해야 합니다.

### Level 1 (Read Only)

가장 안전한 단계

* DB 상태 조회
* 성능 분석
* Slow query 분석
* 인덱스 추천
* Vacuum 상태 분석

예

```
User:
왜 이 쿼리가 느려?

Agent:
EXPLAIN ANALYZE 실행
-> 실행 계획 분석
-> Index 추천
```

---

### Level 2 (Safe Operations)

제한된 변경 작업

* Index 생성
* Autovacuum 튜닝
* Query rewrite
* Statistics refresh

---

### Level 3 (Dangerous Operations)

아래 작업은 반드시 **Human Approval** 필요

* DROP TABLE
* DELETE
* UPDATE
* ALTER TABLE

---

# 4. SQL Safety Guard (가장 중요)

DBA Agent에서 가장 중요한 부분입니다.

### SQL Risk Classifier

SQL을 위험도별로 분류합니다.

| Level     | Query        |
| --------- | ------------ |
| SAFE      | SELECT       |
| WARNING   | CREATE INDEX |
| DANGEROUS | UPDATE       |
| CRITICAL  | DROP         |

---

### Guard Layer

```
LLM SQL
 ↓
SQL Parser
 ↓
Risk Classifier
 ↓
Policy Engine
 ↓
Execution
```

예

```
DROP TABLE users
```

→ BLOCK

또는

```
DELETE FROM logs WHERE date < now() - interval '1 year'
```

→ Approval Required

# 5. Query Simulation Layer

DBA Agent는 반드시 **실행 전에 영향 분석**을 해야 합니다.

### PostgreSQL 기능 활용

```
EXPLAIN
EXPLAIN ANALYZE
pg_stat_statements
pg_locks
pg_stat_activity
```

예

Agent workflow

```
1 User query
2 LLM SQL 생성
3 Harness SQL Validation
4 EXPLAIN 실행
5 영향 분석
6 승인
7 실제 실행
```

 ↓

Query Pattern

# 14. DBA Agent 핵심 차별화

일반적인 NL2SQL과 차이

| 일반 NL2SQL | DBA Agent  |
| --------- | ---------- |
| SQL 생성    | DBA 업무 수행  |
| SELECT 중심 | 운영 자동화     |
| 단순 질문     | 분석 + 해결    |
| LLM 중심    | Harness 중심 |

---

💡 **중요한 인사이트**

요즘 DBA Agent 구조는

```
LLM 20%
Harness 80%
```

입니다.

LLM보다 **Execution Control Architecture**가 핵심입니다.
