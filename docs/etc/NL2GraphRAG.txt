PostgreSQL 기반 **NL2SQL 시스템을 실제 제품 수준으로 만들 때** 요즘은 단순 RAG가 아니라 **GraphRAG 기반 아키텍처**로 설계하는 경우가 많습니다.
특히 **대규모 스키마(수백~수천 테이블)** 환경에서는 GraphRAG이 거의 표준 아키텍처로 가고 있습니다.

아래는 **PostgreSQL 전용 NL2SQL + GraphRAG 실제 구현 아키텍처**입니다.

---

# 1. 왜 NL2SQL → GraphRAG으로 이동하는가

기존 NL2SQL 방식은 다음 문제가 있습니다.

### 1️⃣ 스키마 규모 문제

기업 DB

```
tables: 300~2000
columns: 5000~20000
```

LLM Prompt에 넣을 수 없음.

그래서 대부분 NL2SQL 실패 원인은

```
schema selection 실패
```

입니다.

---

### 2️⃣ 테이블 관계 이해 실패

예

```
orders
customers
payments
shipment
```

질문

```
"지난달 배송 완료 매출"
```

필요 SQL

```
orders
JOIN payments
JOIN shipment
```

LLM이 **join path를 모르는 경우가 많습니다.**

---

### 3️⃣ Business semantic 문제

예

```
active_customer
vip_customer
cancel_rate
```

이건 단순 SQL이 아니라

```
business metric
```

입니다.

---

## 그래서 등장한 접근

**GraphRAG**

핵심 아이디어

```
DB Schema → Knowledge Graph 변환
```

LLM이

```
Graph reasoning
```

으로 SQL 생성.

---

# 3. PostgreSQL Schema → Graph 생성

가장 중요한 단계입니다.

PostgreSQL metadata를 그래프로 변환합니다.

PostgreSQL catalog

```
pg_tables
pg_class
pg_attribute
pg_constraint
```

여기서 관계 추출

---

## Graph Node

```
Table
Column
Metric
Business Concept
```

예

```
Table: orders
Column: order_id
Column: customer_id
Column: order_date
```

---

## Graph Edge

```
table → column
table → table (foreign key)
metric → column
business_term → metric
```

예

```
orders.customer_id
      ↓
customers.id
```

---

## Graph Example

```
customer
   ↑
orders
   ↓
payments
```

또는

```
metric: monthly_revenue
      ↓
orders.amount
```

# 6. Query Planner (매우 중요)

Graph 기반 query plan 생성

예

질문

```
"지난달 매출 상위 고객"
```

Plan

```

2 join customers
3 filter last month
4 group by customer
5 sum revenue
6 order desc
```

---

# 7. SQL Generator

Planner 결과를 SQL로 변환

예

```
SELECT
    c.customer_name,
    SUM(o.amount) as revenue
FROM orders o
JOIN customers c
ON o.customer_id = c.id
WHERE o.order_date >= date_trunc('month', now()) - interval '1 month'
GROUP BY c.customer_name
ORDER BY revenue DESC
LIMIT 10
```

# 13. 요즘 NL2SQL 제품들이 GraphRAG으로 가는 이유

실제 산업에서는

```
LLM + RAG
```

만으로는

```
complex schema
```

를 해결 못합니다.

그래서

```
semantic graph layer
```

가 필요합니다.

최근 시스템 구조

```
Natural Language
      ↓
Semantic Graph
      ↓
Query Plan
      ↓
SQL
```
